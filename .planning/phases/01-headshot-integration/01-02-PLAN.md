URL contract (locked by 01-CONTEXT.md D-02):
- Env var: `SCHOLARS_HEADSHOT_BASE`
- Default base: `https://directory.weill.cornell.edu/api/v1/person/profile`
- Full URL template: `${BASE}/${cwid}.png?returnGenericOn404=false`
- The `false` in `returnGenericOn404` is INTENTIONAL and DIFFERENT from PubMan (which uses `true`). See 01-RESEARCH.md Pitfall 2 — PubMan's `=true` means the directory always returns an image, defeating the fallback. We MUST use `=false` so the directory returns an HTTP 404 and Radix's fallback kicks in.

`data-headshot-state` contract (locked by 01-UI-SPEC.md Visual States):
- `loading` — image fetch in flight (Radix `loading` status, or `idle` with src set)
- `image` — image loaded successfully (Radix `loaded` status)
- `fallback` — image error (Radix `error` status) OR empty `identityImageEndpoint` OR empty `cwid`
- The attribute is on the OUTERMOST `Avatar` root element (UI-SPEC §"Visual States" final paragraph).

**File 1: Create directory and component `components/scholar/headshot-avatar.tsx`** (new file). Use this exact content:

```tsx
"use client";

import { useState } from "react";
import Image from "next/image";
import { Avatar, AvatarImage, AvatarFallback } from "@/components/ui/avatar";
import { cn, initials } from "@/lib/utils";

type HeadshotState = "loading" | "image" | "fallback";

const SIZE_CLASS: Record<"sm" | "md" | "lg", string> = {
  sm: "size-6",
  md: "h-12 w-12",
  lg: "h-24 w-24 sm:h-28 sm:w-28",
};

const FALLBACK_TEXT_CLASS: Record<"sm" | "md" | "lg", string> = {
  sm: "text-xs",
  md: "text-sm",
  lg: "text-xl",
};

export function HeadshotAvatar({
  cwid,
  preferredName,
  identityImageEndpoint,
  size,
  className,
}: {
  cwid: string;
  preferredName: string;
  identityImageEndpoint: string;
  size: "sm" | "md" | "lg";
  className?: string;
}) {
  const [imgStatus, setImgStatus] = useState<"loading" | "loaded" | "error">(
    "loading"
  );
  const noImage = !identityImageEndpoint || !cwid;

  const dataState: HeadshotState = noImage
    ? "fallback"
    : imgStatus === "loaded"
      ? "image"
      : imgStatus === "error"
        ? "fallback"
        : "loading";

  return (
    <Avatar
      data-headshot-state={dataState}
      className={cn(SIZE_CLASS[size], "shrink-0", className)}
    >
      {!noImage && (
        <AvatarImage
          asChild
          onLoadingStatusChange={(s) =>
            setImgStatus(
              s === "loaded" ? "loaded" : s === "error" ? "error" : "loading"
            )
          }
        >
          <Image
            src={identityImageEndpoint}
            alt={preferredName}
            fill
            unoptimized
            sizes="(max-width: 640px) 96px, 112px"
          />
        </AvatarImage>
      )}
      <AvatarFallback className={FALLBACK_TEXT_CLASS[size]}>
        {initials(preferredName)}
      </AvatarFallback>
    </Avatar>
  );
}
```

Notes:
- The component is a client component (D-06 requires `useState`).
- `data-headshot-state` lives on the outermost `Avatar` element per UI-SPEC.
- `noImage` defensively guards both empty `identityImageEndpoint` AND empty `cwid` (UI-SPEC § "no-cwid" state collapses into `fallback`).
- The `asChild` + `next/image` pattern is the primary attempt. If integration testing in Wave 4 reveals that `onLoadingStatusChange` never fires `"loaded"` in production (Pitfall 1), a follow-up plan can swap to a plain `<img>`. The unit test in jsdom only ever observes the initial `"loading"` state, so this concern does not block this task.
- `text-sm` for md is added because the existing search row had no explicit text-size override; the existing profile sidebar already passed `className="text-xl"` on AvatarFallback — `lg` keeps that token.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → directory.weill.cornell.edu | Browser fetches the headshot image directly. The boundary is between the user's browser and an external WCM-org HTTPS service. No data flows from the user other than the GET request URL containing the CWID. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-01 | Information Disclosure | next.config.ts remotePatterns | mitigate | Pin hostname to `directory.weill.cornell.edu` exactly; do not use a wildcard subdomain. The whitelist restricts what `next/image` will render — even if `identityImageEndpoint` were tampered with, only directory.weill.cornell.edu URLs render. |
