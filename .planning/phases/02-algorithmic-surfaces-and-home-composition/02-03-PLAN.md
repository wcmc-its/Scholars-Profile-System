<objective>
Wave 0 prerequisite. The eligibility carve (Full-time faculty + Postdoc + Fellow + Doctoral student) cited everywhere in design spec v1.7.1, ROADMAP, CONTEXT.md, and RESEARCH.md has NO data feed today. `etl/search-index/index.ts:120` hard-codes `personType = "Faculty"` as a placeholder; ED ETL pulls `weillCornellEduPersonTypeCode` but does not project it; doctoral students live under `ou=students` (not `ou=people`) and the current `(weillCornellEduPersonTypeCode=academic)` filter excludes them entirely. Without this plan, every Phase 2 surface query that filters on `roleCategory` (RANKING-01, RANKING-03 D-14, HOME-02 carousel filtering) will fail or return empty.

This plan:
1. Adds `scholar.role_category` Prisma column + migration.
2. Creates `lib/eligibility.ts` with `RoleCategory` enum-as-string-union, `ELIGIBLE_ROLES` (the four-role general carve), and `TOP_SCHOLARS_ELIGIBLE_ROLES` (the FT-faculty-only carve per CONTEXT.md D-14).
3. Extends `lib/sources/ldap.ts` to pull `weillCornellEduFTE` and `weillCornellEduDegreeCode`, and adds `fetchDoctoralStudents` for the `ou=students` branch.
4. Extends `etl/ed/index.ts` with `deriveRoleCategory` and the second LDAP query branch; writes `role_category` on every insert/update.
5. Replaces the `personType = "Faculty"` hard-code in `etl/search-index/index.ts` with a read from `scholar.roleCategory`.
6. Adds `tests/unit/eligibility.test.ts` so the constants are guarded against drift.

Purpose: Wave 1 (Plan 04) and Wave 3 (Plans 07, 08) all depend on `ELIGIBLE_ROLES` and `TOP_SCHOLARS_ELIGIBLE_ROLES` being importable, and on `scholar.role_category` being populated. RESEARCH.md §Pitfall 1 (CRITICAL) flags this as the most likely cause of Phase 2 failure if skipped.

Output: Migration applied, ED ETL writes role_category for all 8,943+ active scholars + adds doctoral students, search-index ETL fixed, eligibility module + tests in place.
