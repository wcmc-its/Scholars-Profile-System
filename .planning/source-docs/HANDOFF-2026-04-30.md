# Scholars Profile System — Handoff Document

_Last updated: 2026-04-30_
_Audience: Mahender, Mohammad's team, AAC, CAB_

## What this is

A working **local prototype** of the Phase 1 Scholars @ WCM system — the planned VIVO replacement. Built end-to-end against real WCM data sources (ED, ReciterDB, ASMS, InfoEd, COI, ReciterAI DynamoDB) so you can see actual scholar profiles, search results, grants, disclosures, and publications without any synthetic placeholders.

This is a **reference implementation**, not a production deploy. AWS infrastructure, the production technical plan, and the official design decisions are owned by Mohammad's team. Use this prototype to:
- React to the architectural decisions before the production build kicks off
- Validate UX / feature scope / coverage assumptions
- Borrow code, schemas, ETL queries, or skip everything and start fresh

## Current state — what works

- **8,943 active WCM faculty** populated from live ED via LDAP
- **Profile pages** render: header, overview (currently empty for real scholars — overview is a self-edit field), contact (email), appointments, education, areas of interest, publications (with both ranking formulas + WCM coauthor chips), grants, disclosures
- **Search** with per-field boost weights (Name 10× / AOI 6× / Title 4× / Department 3× / Overview 2× / Pub titles 1× / MeSH 0.5×) and authorship-weighted contributions. Faceted results page with people/publications tabs, year-range filter, autocomplete on 2 chars, numbered pagination 20/page.
- **Daily ETL chain** with ED-first abort cascade, per-source independence, and `/api/health/refresh-status` admin endpoint
- **Slug-primary URLs** with collision suffixing and slug-history 301 redirects
- **CWID-canonical identity** with `cwid_aliases` table for replacement-CWID handling
- **Soft-delete** of departed scholars with 60-day retention window

## Eight architectural decisions

Detailed in `Phase 1 Design Decisions - 2026-04-29.md`. Quick summary:

| # | Topic | Decision |
|---|---|---|
| 1 | API architecture | Single Next.js deploy + `/api/*` routes; OpenAPI contract artifact. **Mohammad's preliminary preference is a separate API service in production — deferred to his official design kickoff.** |
| 2 | Identity | CWID-canonical with `cwid_aliases` from ED `replacement_cwid` |
| 3 | URL strategy | Slug-primary HTML + CWID-keyed API + `/scholars/by-cwid/:cwid` fallback |
| 4 | Departed faculty | Strict delete on appointment loss, 60-day soft-delete retention; VIVO URLs for departed → 301 to `/search?q=name` |
| 5 | Daily refresh failure modes | Per-source independent + ED-first abort cascade + monitoring |
| 6 | DynamoDB integration | Minimal-projection ETL (publication_score + topic_assignments only); ReciterDB for publication metadata |
| 7 | Search engine | OpenSearch; dense embeddings deferred to Phase 2 |
| 8 | Render strategy | Next.js ISR with on-demand revalidation; CSR for search via `/api/search` proxy |

**Spec amendment** added during build: COI "Disclosures" section on profile pages. The functional spec didn't enumerate it, but VIVO surfaced this data and removing it would be a regression.

## What's NOT done (build-plan phases 5–7)

- **Phase 5 — SEO + URL machinery.** sitemap.xml generation, robots.txt, full VIVO 301 mapping. Currently the URL middleware does slug→cwid→alias resolution but the sitemap and bulk VIVO 301 audit aren't wired.
- **Phase 6 — Polish, analytics, docs.** Page-view tracking, search-query logging, OpenAPI artifact, the Q6 ADR (DAL = ETL transform). The functional spec calls for several analytics signals (line 277-283) and we haven't instrumented them.
- **Phase 7 — Self-edit + auth.** SAML wiring, `/api/edit` route, WYSIWYG, "What's missing" checklist for authenticated owners, edit-event logging.

## How to run locally

Prereqs: Node 22+, Docker, npm 10+, VPN access for source systems, env vars below in `~/.zshenv`.

```bash
# 1. Clone and install
git clone https://github.com/wcmc-its/Scholars-Profile-System.git
cd Scholars-Profile-System
npm install

# 2. Start local DBs (MySQL + OpenSearch)
npm run db:up

# 3. Apply schema
npx prisma migrate dev

# 4. Seed synthetic data (12 fictional scholars for testing without VPN)
npm run seed

# 5. OR: pull real WCM data via the daily ETL chain
npm run etl:daily          # ~10 min; needs VPN + env vars

# 6. Build search index
npm run search:index

# 7. Start dev server
npm run dev
```

Visit `http://localhost:3000`.

## Lessons learned during the build (worth knowing for production)

1. **Prisma 7 broke the embedded engine model.** Driver adapter required (`@prisma/adapter-mariadb` works for both MySQL 8 and MariaDB). The Aurora MySQL production target uses the same adapter.

2. **Local MySQL needs CREATE/DROP at the global level** for Prisma's shadow-database migration drift detection. In production, this should be a dedicated migration user; the runtime app user stays scoped to its own database.

3. **ReciterDB `personIdentifier` is plain CWID — no `cwid_` prefix.** The `cwid_` prefix in the ReciterAI integration project is **DynamoDB-only** (PK uses `FACULTY#cwid_<cwid>`). I burned an iteration on this because the CLAUDE.md note about WCM_FACULTY_UID_PREFIX was DynamoDB-specific and I assumed ReciterDB shared the convention.

4. **InfoEd's full institutional grant query is a 30-table join across DBs and runs ~6 minutes.** `requestTimeout` needs to be 10+ minutes. In production, this should be batched, cached, or rewritten as a materialized view in the InfoEd-side warehouse.

5. **COI is on a MySQL RDS instance, not MSSQL.** The hostname (`*-mysql-db.*`) gave it away on the first connection failure. Worth being explicit in any new ETL inventory.

6. **Several password formats had to be unwound.** Some env vars came from K8s Secret YAML blocks (base64-encoded, YAML-folded), some had `$` characters that triggered shell variable expansion in double-quoted exports. Production should rely on AWS Secrets Manager / SSM Parameter Store rather than shell exports.

7. **Mohammad's API-service preference logged.** During the build, Mohammad sent a preliminary preference for a separate Scholar API service in production. The prototype implementation discipline (handlers as pure functions in `lib/api/*`, route files as thin delegators) makes lifting handlers into a standalone service a copy-paste, not a rewrite. The OpenAPI contract — once written — is the durable artifact regardless of implementation choice.

## What I'd recommend for the production handoff

1. **Read `Phase 1 Design Decisions - 2026-04-29.md` first** — eight calls with rationale, including ones that are reversible if Mohammad's team decides differently.

2. **The eight ETLs in `etl/<source>/`** are the most reusable parts of this prototype. Even if production uses a different framework, the SQL queries, MSSQL/LDAP connection patterns, and authorship-position classification logic are useful as starting templates.

3. **The `analysis_summary_*` join shape** for ReciterDB is documented in `etl/reciter/index.ts`. The institutional client uses `wcmc_*` legacy tables which don't exist in your current ReciterDB; this prototype uses the modern path.

4. **The search index strategy** (per-field boost at query time, authorship weighting at index time via term repetition) is a deliberate choice. See decision #7 and `etl/search-index/index.ts`. If Mohammad's team picks a different search engine, the index-time term-repetition pattern can travel.

5. **What's NOT in scope for the handoff:** AWS provisioning, CAB/AAC approval artifacts, production security review, data privacy / FERPA / HIPAA review for COI display. Those are Mohammad's team's lane.

## Open items requiring stakeholder input

- **Confirm ED `preferred_name` field** as the slug source (vs `cn` / `givenName + sn`). Currently slugs are derived from `givenName + sn`; if WCM has a separate preferred-name system, the slug source should switch.
- **AOI threshold + 6× search boost calibration** (spec line 313). We took ReciterAI's top_topics as-is; production should calibrate against real search behavior.
- **Publication ranking weight calibration** against ~20 real WCM profiles spanning seniority (spec line 314). Highlighted in spec as "where ranking errors generate angry emails."
- **Citation refresh cadence in reciterdb-prod** (spec line 319). Spec target is ≥ weekly.
- **Edit-event logging target** (spec line 316) — Slack? Email digest? Decided by end of design phase per spec; Phase 7 work.
- **Specific completeness threshold** for "being populated" affordance and default-search filtering.
