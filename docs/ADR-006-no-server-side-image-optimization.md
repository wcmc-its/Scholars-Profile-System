# docs/ADR-006 — No server-side image optimization

**Status:** Accepted
**Date:** 2026-05-18
**Authors:** Scholars Profile System development team
**Supersedes:** —
**Superseded by:** —
**Resolves:** [#114](https://github.com/wcmc-its/Scholars-Profile-System/issues/114) — B15, `next/image` runtime cost decision (part of [#99](https://github.com/wcmc-its/Scholars-Profile-System/issues/99) production readiness)

---

## Context

B15 ([#114](https://github.com/wcmc-its/Scholars-Profile-System/issues/114)) raised a production-readiness concern: `next/image` optimizes images at request time with `sharp`, which is CPU-heavy, and on a 1-vCPU Fargate task — Prisma `connection_limit=15`, per [`PRODUCTION.md`](./PRODUCTION.md) — that optimization would contend with the connection pool and "starve Prisma". It asked for a decision between (a) pre-resizing images at ETL time and serving them from S3 + CloudFront, (b) `images.unoptimized: true`, or (c) a separate image-optimization service.

A 2026-05-18 review of the codebase found the concern's premise does not hold:

- **`next/image` is used nowhere.** Source search returns zero `from "next/image"` imports and zero `<Image>` components. Its request-time optimizer — the `sharp`-backed `/_next/image` endpoint — is reached only by `<Image>`, so it is never invoked.
- **Headshots — the app's only photographic content — render as a plain `<img>`.** `components/scholar/headshot-avatar.tsx` wraps a Radix UI `<Avatar>`. The `src` is an external URL built by `lib/headshot.ts`: `https://directory.weill.cornell.edu/api/v1/person/profile/{cwid}.png`, already a fixed-size PNG. The browser fetches it directly; the Next.js server is never in the image byte path.
- **No ETL step ingests, resizes, or stores images.**
- **Open Graph card images** are generated by `next/og` `ImageResponse` (`app/og/scholars/[slug]/route.tsx`) — a distinct subsystem, unrelated to `next/image` request-time optimization.
- The site has no raster logo or hero images; branding is text and CSS.

The request-time `sharp` cost B15 is concerned about therefore is not, and cannot be, incurred. Options (a), (b), and (c) all address a cost that does not exist.

The `next/image` assumption had spread beyond the issue: `next.config.ts` carried an `images.remotePatterns` allowlist for `directory.weill.cornell.edu` (never consulted — no `<Image>` exists to consult it), and comments in `lib/security-headers.ts` and `tests/unit/headshot-avatar.test.tsx`, plus `docs/PRODUCTION_ADDENDUM.md`, all described image handling as if `next/image` were in use.

## Decision

The Scholars Profile System performs **no server-side image optimization**. This ADR records that as the deliberate default, not an accident of current code.

- `next.config.ts` sets `images: { unoptimized: true }` — option (b) from B15. With `next/image` unused it has no runtime effect today; its purpose is a **guardrail**. If an `<Image>` component is ever added, Next.js will serve the original bytes instead of invoking `sharp` on the 1-vCPU Fargate task. The decision is then enforced by configuration, not by reviewer vigilance.
- The dead `images.remotePatterns` allowlist is removed. It gated the unused `next/image` optimizer; keeping it implied an `<Image>` integration that does not exist.

Headshots continue to render as plain `<img>` elements (Radix Avatar) loaded directly from the WCM directory API. In effect this is stronger than option (b): the server does not merely skip optimization, it is not in the image request path at all.

## Consequences

### Positive

- The B15 Fargate/`sharp` CPU-contention risk is closed by inspection — it cannot occur.
- `images: { unoptimized: true }` makes "no server-side image optimization" explicit and durable; a later `<Image>` cannot silently re-introduce request-time `sharp`.
- `next.config.ts` no longer carries dead, misleading `next/image` configuration.

### Negative / accepted

- `images.unoptimized: true` is a global switch. If the project later wants genuine `next/image` optimization (responsive `srcset`, AVIF/WebP), a developer must consciously remove the flag and add an appropriate `remotePatterns` allowlist — and, per B15's original concern, first decide how `sharp` cost is bounded on the Fargate task. This ADR is the marker for that decision; the correct path is to supersede it, not to quietly delete the flag.
- No build-time AVIF/WebP conversion or responsive `srcset` is produced. Accepted: the only photographic asset is the headshot — already a small fixed-size PNG from an external API, displayed at fixed CSS sizes — for which responsive variants would add nothing.

### Operational

- B15's acceptance criterion "load test in staging confirms image requests don't starve Prisma under 100 RPS" is moot: there are no server-side image requests to load-test. It is recorded here as discharged, not deferred.

## Alternatives considered

- **(a) Pre-resize headshots at ETL → S3 + CloudFront with a `next/image` `loader`.** Rejected: it presumes `next/image` is the renderer and that the app stores headshots — neither is true. Headshots are fetched live, browser-side, from the WCM directory API. This would be net-new infrastructure (an ETL image step, an S3 bucket, a CloudFront origin, a custom loader) to optimize bytes the server never handles.
- **(c) A separate image-optimization service.** Rejected for the same reason at higher cost; B15 itself flagged it as not recommended at this scale.
- **Leave `next.config.ts` as-is.** Rejected: the dead `remotePatterns` and the inaccurate surrounding comments actively mislead — they imply a `next/image` integration that does not exist. Recording the decision and removing the dead config has lasting value even though immediate runtime behavior is unchanged.

## Non-goals

- **How headshots are _delivered_.** This ADR concerns server-side _optimization_ only. A separate, real question — outside B15's scope — is that headshots load browser-side directly from `directory.weill.cornell.edu` on every render: no app-side CDN caching, and no fallback if that API is slow or unavailable (the component degrades to an initials placeholder — acceptable, but unmonitored). If that is a production-readiness gap it warrants its own issue; it is not decided here.
- **The CSP `img-src` directive.** `lib/security-headers.ts` keeps `img-src 'self' data: blob: https://directory.weill.cornell.edu` unchanged; this ADR corrects only the stale comment beside it. CSP tuning is [#374](https://github.com/wcmc-its/Scholars-Profile-System/issues/374)'s scope.
- **`docs/PRODUCTION_ADDENDUM.md` and `docs/PRODUCTION_BACKLOG.md`.** Both predate this finding and still describe images in `next/image` terms. They are planning documents superseded by this ADR and are not edited here.
