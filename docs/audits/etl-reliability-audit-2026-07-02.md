# ETL Reliability Audit — 2026-07-02

> **Snapshot as of 2026-07-02 — point-in-time; not maintained.** Kept for provenance under [Audits & snapshots](../DOCUMENTATION-REGISTRY.md).

**Scope:** every deployed ETL module (nightly / weekly / annual / heartbeat cadences) plus the
Step Functions machine definitions, audited for destructive/non-idempotent write patterns,
partial-source hazards, re-run safety, error swallowing, and observability gaps.

**Provenance:** audited at `origin/master` @ `baca61e0` (2026-07-02) in a dedicated read-only

The guardrail PRs below should be treated as prerequisites for that flip.

## Executive summary

The pipeline's dominant design is **mirror-the-source**: fetch a source, then delete/tombstone
whatever SPS holds that the source no longer returns. That design is correct when the source is
healthy and catastrophic when a source returns a *successful but truncated* result — and not one
module in the deployed cadences has a suspiciously-few-rows guard in front of its destructive
pass. Four confirmed-critical paths let a single silently-degraded upstream (LDAP ACL scope
change, ReciterDB mid-rebuild read, empty COI view) gut user-visible data — scholars, the ~178k
publication spine, or all COI disclosures — while the step exits 0, `etl_run` records SUCCESS,
the freshness heartbeat stays green, and the closing `search:index` rebuild + ISR revalidate
publish the gutted state to the site.

Five recurring patterns account for nearly all 67 findings:

1. **Unguarded mass-delete on partial/empty source** (~25 findings, incl. all 4 criticals).
   No floor/delta check anywhere between "source responded" and "delete what it didn't mention."
   The repo already contains the right pattern — `guardedReplace` in
   `etl/dynamodb/projection-replace.ts` — it just isn't used anywhere else.
2. **Non-transactional delete-then-repopulate** (COI, Tools ×4 tables, PublicationAuthor,
   clinical-trials, Jenzabar): a mid-step kill (OOM, deploy, SIGKILL) leaves a table empty or
   partial, and the Step Functions retry (whole-step re-run ×2) *widens* the window instead of
   healing it.
3. **Post-swap verification in search:index**: smoke assertions run *after* the alias swap, and
   there is no old-vs-new doc-count comparison — an empty or halved index goes live before
   anything can object.
4. **Green-but-broken observability**: `EtlRun.rowsProcessed` is written by every module and
   read by nothing; the freshness heartbeat checks recency only and its SLA table has drifted
   from the actual cadences (Jenzabar tracked weekly but runs nightly; InfoEd tracked but
   excluded from staging; the whole weekly enrichment tier + terminal steps invisible; five
   sources in no deployed cadence at all).
5. **Abort-on-first-failure divergence**: the deployed machine fails the entire chain on any
   step failure (aborting the closing `search:index` + revalidate), while the design intent in
   `etl/orchestrate.ts` explicitly isolates per-source failures after the ED chain head.

**Empirical operational finding (verified via AWS CLI, not code):**
`etl-failures-staging` is subscribed by `sps-oncall-relay-staging` (Lambda), but
**`etl-failures-prod` has ZERO subscriptions** — a prod ETL failure notification publishes to
nobody. This must be wired (mirror the staging relay, or at minimum an email subscription to
the on-call address) **before** prod cadence schedules are enabled.

## Hardening plan (proposed PRs, in value order)

**OPS-0 — subscribe `etl-failures-prod` (no code).** Mirror `sps-oncall-relay-staging` for prod
or add an email subscription. Prerequisite for enabling prod schedules. Do immediately.

**PR-1 — shrink guard on the four critical paths.** Add `lib/etl-guard.ts` (generalize the
existing `guardedReplace` floor/delta logic): `assertSourceVolume(source, incomingCount,
existingCount, { floor, maxDropPct })` that **throws** (existing Catch→SNS path handles the
rest) unless an explicit env override is set. Apply to: ED faculty feed + appointments feed +
doctoral-student swallow (`etl/ed/index.ts:871,240,379`), Reciter orphan-prune
(`etl/reciter/index.ts:568`), COI wipe (`etl/coi/index.ts:64`, also scope the delete to
`source:"COI"`). Small diff, kills all four criticals.

**PR-2 — search:index pre-swap gate.** Per-index old-vs-new doc-count delta check against the
live alias + move the smoke assertions before the swap; abort (no swap) on failure — the stale
index stays live, which is strictly better than an empty one.

**PR-3 — extend the shrink guard to every remaining destructive writer.** InfoEd grants,
Tools 4-table wipe, reporter `grant_publication`, Jenzabar (+ gs-faculty import), ASMS,
clinical-trials direct entrypoint, family-sensitivity/suppression/review + mesh-aliases/anchors
(treat curated-CSV ENOENT as abort, not empty), headshot mass-flip cap, NYP empty-success,
student-programs, reciter secondary sources (abstracts/author-strings/mesh nulling) and
analysis_nih, reporter-grants (exclude errored scholars from the prune set), POPS zero-success
floor.

**PR-4 — freshness/etl_run truth sync.** Fix SLA drift (Jenzabar → nightly SLA; InfoEd
staging exclusion mirrored in TRACKED), add `etl_run` rows for ReciterCoiStatements,
SearchIndex, Revalidate, clinical-trials direct entrypoint; add the weekly tier to TRACKED;
decide fate of the five orphaned sources in no cadence (POPS, Reporter-Grants, Clinical-Trials,
ED-Student-Programs, ED-Admins) — schedule or explicitly retire.

**PR-5 — post-ETL integrity gate.** New terminal nightly step `etl:integrity`: compare each
source's `rowsProcessed` vs its previous success (>50% drop ⇒ fail), assert table-count floors
for scholar/publication/authorship/coi_activity/grant, named canaries (e.g. `chm2042`
publicationCount > 0 — also the post-VPC-cutover canary), OpenSearch alias doc-count vs Aurora
count. Exits non-zero → existing alarm path. The inputs already exist (`rowsProcessed` is
written by every module today and read by nothing).

**PR-6 — transactionality for the worst delete-rebuild windows.** COI wipe→insert in one
`$transaction` (or staged swap table), Tools 4-table wipe, clinical-trials `replaceAll`,
Jenzabar truncate, PublicationAuthor per-PMID batched delete+reinsert.

**PR-7 — state-machine semantics (cdk).** (a) Classify steps continue-vs-abort: keep abort for
Ed/Reciter/Dynamodb (spine + dependents), Catch→notify→**continue** for isolatable sources
(Asms, Jenzabar, Identity, Tools, CoiGap, MeshCoverage, PubMedRetractions) so one flaky source
stops costing the nightly index rebuild + revalidate; (b) `startFrom` strict validation (fail
on unmatched id instead of silently running the full chain); (c) Sunday nightly/weekly overlap
guard (weekly waits on nightly execution, or move weekly later + add a concurrency check);
(d) per-step timeouts sized to observed durations instead of the 24h blanket.

Suggested sequencing: OPS-0 now → PR-1 → PR-2 → PR-4+PR-5 → PR-3 → PR-6 → PR-7.

## Confirmed findings (adversarially verified)

### [CRITICAL] No suspiciously-few-rows guardrail on the ED faculty feed before the mass soft-delete pass — a silently truncated LDAP result guts the whole site

- **Where:** `etl/ed/index.ts:871` · category: partial-source
- **Verification:** CONFIRMED (2 independent adversarial verifiers, 2 affirmed)
- **Failure scenario:** fetchActiveFaculty (lib/sources/ldap.ts:211) returns whatever the bind DN can see under the configured filter. If an ACL scope change on the service account, a bad SCHOLARS_LDAP_ACTIVE_FILTER env value, or a base-DN restructure makes the search return SUCCESS with e.g. 50 rows instead of ~8,900, the soft-delete pass (etl/ed/index.ts:871-881) marks every missing scholar deletedAt=now() with no minimum-count or max-delta check anywhere. The Ed step then exits 0, etl_run records SUCCESS, the rest of the nightly runs, and the closing SearchIndex rebuild + Revalidate publish the gutted state: nearly every profile hidden, people index near-empty. The same run's scholarCount refresh then computes 0 for most departments, triggering the hard prune (see separate finding). The code itself acknowledges silent ACL truncation as a real WCM failure mode (the warning at etl/ed/index.ts:1162-1169 exists precisely because the bind DN can be silently scoped). The same gap applies to fetchActiveFacultyAppointments — a truncated-but-successful appointment feed tombstones every scholar's ED appointments via classifyByExternalId (etl/ed/index.ts:240-244) and clears every department chairCwid (etl/ed/index.ts:1024-1027).
- **Blast radius:** Entire public site: all scholar profiles, people index, search results (reindexed at chain end), department/division pages. Persists until the upstream ACL/filter cause is manually fixed — soft-deleted scholars only reactivate when the feed recovers. Nobody is alerted because the step and etl_run both report success.
- **Recommendation:** Before the soft-delete loop, abort (throw) if facultyEntries.length is below an absolute floor (e.g. 5,000) OR if departed.length exceeds a small percentage of the existing active set (e.g. >2%), requiring an explicit env override to proceed. Apply the same floor/delta check to fetchActiveFacultyAppointments before refreshEdAppointments/chair clearing. A thrown error correctly aborts the nightly via the existing Catch->SNS path.

### [CRITICAL] Orphan-publication cleanup has no minimum-row guardrail: an empty/truncated ReciterDB source cascades-deletes the entire ~178k publication spine and records SUCCESS

- **Where:** `etl/reciter/index.ts:568` · category: partial-source
- **Verification:** CONFIRMED (2 independent adversarial verifiers, 2 affirmed)
- **Failure scenario:** ReciterDB is reachable but analysis_summary_author (or analysis_summary_article) is empty or truncated — e.g. the upstream ReciterDB nightly rebuild failed mid-truncate, or the SPS nightly overlaps its rebuild window. Queries succeed with few/zero rows, so authorRows→distinctPmids→pubRows shrink (a pmid missing from analysis_summary_article silently drops out of pubRows at line 480). sourcePmidsSet is now tiny, and the orphan cleanup (lines 564-579) computes orphanPmids = ALL existing pmids not in that set and runs publication.deleteMany in IN-batches (line 576), cascading (per the header comment, lines 17-19) into publication_topic (owned by the dynamodb ETL), publication_author, grant_publication, publication_score, and publication_conflict_statement. No 'suspiciously few rows' check exists anywhere between the fetch and the delete. The run then writes etl_run status=success (line 584), the later SearchIndex step rebuilds and atomically alias-swaps an index over the gutted DB, and Revalidate publishes it — so both DB-rendered profiles and search show near-zero publications the next morning.
- **Blast radius:** Site-wide: every scholar profile's publications, topics (via publication_topic cascade), grant-publication links, and the search index are gutted until an operator restores the upstream source and re-runs etl:reciter + etl:dynamodb + funding reconcile + search:index. Both the Step Functions machine and the etl_run freshness heartbeat report green throughout.
- **Recommendation:** Before the orphan delete, abort (throw, so the step fails and SNS fires) when authorRows/pubRows fall below a floor relative to the current publication count (e.g. pubRows < 0.8 * existingPmids.length) or when orphanPmids exceeds a small fraction (e.g. >5%) of existing rows; optionally compare against the previous successful etl_run.rowsProcessed. A hard cap on orphan deletions per run is the cheapest effective guard.

### [CRITICAL] COI ETL wipes all disclosures with no minimum-row guard; an empty/truncated source view is mirrored as a full wipe recorded as SUCCESS, and the next step fabricates false compliance gaps

- **Where:** `etl/coi/index.ts:64` · category: partial-source
- **Verification:** CONFIRMED (2 independent adversarial verifiers, 2 affirmed)
- **Failure scenario:** The source query (etl/coi/index.ts:48-57) succeeds but returns 0 or few rows (upstream v_coi_vivo_activity_group truncated mid-refresh on the COI RDS, view definition drift, credentials pointing at a restored-empty DB). There is no suspiciously-few-rows check: line 64 runs an unscoped `coiActivity.deleteMany()` and inserts whatever came back, then records etl_run status=success with rowsProcessed=0 (lines 90-93), so the freshness heartbeat stays green. Because CoiGap runs immediately after Coi in the nightly machine, `computeScholarGaps` reads the now-empty disclosed set (lib/coi-gap/compute.ts:58-64) and every previously-disclosed entity in a conflict statement surfaces as a gap: brand-new rows are inserted as status "new" and previously "resolved" gaps are reopened to "new" (lib/coi-gap/lifecycle.ts:70-73), with the reopen clearing the scholar's prior reviewedAt/feedbackReason (etl/coi-gap/index.ts:175).
- **Blast radius:** Every scholar's Disclosures section is emptied site-wide, AND scholars receive false "undisclosed conflict" nags — a compliance-governance problem, not just UX. Both ETL steps and the heartbeat report green, so nothing pages; the damage persists (and gap-lifecycle history stays polluted with spurious reopen/resolve churn and cleared review metadata) until the upstream source is fixed and a run reloads.
- **Recommendation:** Before the deleteMany, abort (throw, so Step Functions retries/fails and etl_run records failed) when `filtered.length` is 0 or below a sanity floor relative to the current coi_activity row count (e.g. <50% of existing). Optionally scope the delete to `where: { source: "COI" }` so the step only deletes what it owns.

### [CRITICAL] Reciter orphan-prune of publications has no shrink guard and no alarm for partial-source truncation; total-wipe alarms only after the empty search index is already live

- **Where:** `etl/reciter/index.ts:576` · category: observability
- **Verification:** CONFIRMED (2 independent adversarial verifiers, 2 affirmed)
- **Failure scenario:** ReciterDB's analysis_summary_author/analysis_summary_article (themselves rebuilt nightly upstream) return a silently truncated set (partial upstream load, auth scope change, schema drift). Step 7 (etl/reciter/index.ts:566-580) computes orphanPmids = existing DB pmids minus sourcePmids and hard-deletes them, cascading into publication_topic, publication_author, grant_publication, publication_score. etl_run records success with the smaller rowsProcessed (:584-591), the Step Functions nightly stays green, and etl:freshness stays green (it checks recency of success only, never volume). A 50% truncation fires NOTHING anywhere. Only the 0-row extreme is caught — by the search-index smoke (etl/search-index/index.ts:449) — and that fires AFTER the gutted publications index has already been alias-swapped live, and next night by assertPublicationTopicPopulated.
- **Blast radius:** Every scholar profile's publication list, all topic/subtopic pages, publication search, and grant-publication links across the whole site, in whichever env the nightly runs. Deletions are real (cascade), recovered only if the upstream truncation heals and the next nightly re-imports.
- **Recommendation:** Cheapest: apply the guardedReplace pattern already in-repo (etl/dynamodb/projection-replace.ts:80) to the prune — throw before deleting when orphanPmids.length exceeds e.g. 10% of existingPmids.length (or when pubRows.length < 50% of current publication count). Belt-and-braces: extend etl:freshness to compare each source's latest rowsProcessed vs its previous success and alarm on >50% drop (the column is already written, never read).

### [HIGH] PublicationAuthor delete-then-reinsert across all source PMIDs is non-transactional; death between the delete loop and the insert loop leaves every profile with zero publications

- **Where:** `etl/reciter/index.ts:546` · category: partial-failure
- **Verification:** CONFIRMED (1 independent adversarial verifier, 1 affirmed)
- **Failure scenario:** Lines 545-548 deleteMany ALL WCM authorship rows for the full ~178k-pmid source set (batched, no transaction), then lines 550-558 re-insert ~hundreds of thousands of rows via createMany. If the task is OOM-killed / loses the DB between the two loops (or partway through the insert), publication_author is empty or partial. The Step Functions retry re-runs the whole step and does converge (the rebuild is deterministic from source), but if the same underlying fault (Aurora failover, network partition, memory ceiling) also kills both retries, the machine aborts with publication_author gutted — and because SearchIndex/Revalidate are downstream of the Fail, the DB-rendered profile pages (lib/api/profile.ts reads live) show no publications until a manual re-run. Even on a healthy run there is a multi-minute window where profiles render with zero/partial authorships.
- **Blast radius:** All scholar profiles' publication lists and authorship-derived features (first/last-author flags, co-pub chips) empty or partial until a successful re-run; search index stays stale-but-intact (mitigation). Recovery is a single etl:reciter re-run once the fault clears.
- **Recommendation:** Make the swap atomic per pmid-chunk: wrap each chunk's deleteMany+createMany in one $transaction, or better, diff-and-upsert keyed on (pmid, cwid) with a scoped delete of only rows absent from the fresh set — eliminating the empty window entirely.

### [HIGH] COI delete-then-repopulate is not transactional: a kill between the table wipe and the batched inserts leaves coi_activity empty or partial

- **Where:** `etl/coi/index.ts:82` · category: destructive-nonidempotent
- **Verification:** CONFIRMED (1 independent adversarial verifier, 1 affirmed)
- **Failure scenario:** `deleteMany()` (line 64) and the batched `createMany` loop (lines 82-88) run as separate statements with no transaction. If the task dies mid-insert (OOM, SIGKILL, Aurora failover) the table is left with a fraction of disclosures. The step retry does converge (each attempt re-deletes and re-inserts), but if both retries hit the same persistent cause the nightly aborts with coi_activity gutted — and since CoiGap is the next step it never runs against the bad state that night, yet the SITE serves the partial table until a human intervenes. etl_run correctly records failed, so at least the heartbeat sees it.
- **Blast radius:** All scholars' disclosure sections empty/partial until a successful re-run; the abort also cancels every later nightly step including SearchIndex and Revalidate, so the whole site goes stale for the day.
- **Recommendation:** Wrap delete+insert in a single transaction (`db.write.$transaction`, interactive with a raised timeout given ~batched volume), or stage into a shadow table / new rows tagged by runId and swap/delete-old at the end so readers never observe the empty state.

### [HIGH] Tools ETL wipes 4 tables (scholar_tool, scholar_family, family_entity, family_entity_usage) with no transaction around delete->repopulate

- **Where:** `etl/tools/index.ts:571` · category: destructive-nonidempotent
- **Verification:** CONFIRMED (1 independent adversarial verifier, 1 affirmed)
- **Failure scenario:** In s3 mode, scholarTool.deleteMany() (line 571), scholarFamily.deleteMany() (597) and familyEntityUsage/familyEntity.deleteMany() (628-629) each run OUTSIDE any transaction, followed by long chunked createMany loops. A kill (OOM, SIGKILL, DB drop) between a deleteMany and completion of its createMany loop leaves that table empty or partial. Step Functions retries re-run from scratch and normally converge (the failed attempt records status=failed, so the sha short-circuit at line 280-296 does not block the retry), but a DETERMINISTIC failure — a row violating a column constraint, an oversized JSON exemplarContexts value, recurring OOM on the large insert loops — fails all 3 attempts AFTER the wipe, aborting the rest of the nightly (including SearchIndex and Revalidate) and leaving the tables gutted until manual intervention.

The four tables come from one artifact fetch and can share one transaction.

### [HIGH] Search-index smoke assertions run AFTER the alias swap, and an empty corpus read swaps an empty index live with no shrinkage guard

- **Where:** `etl/search-index/index.ts:754` · category: partial-source
- **Verification:** CONFIRMED (1 independent adversarial verifier, 1 affirmed)
- **Failure scenario:** indexPeople/indexFunding/indexOpportunities return 0 without error when their Prisma read yields no docs (index.ts:223, 337, 661) — e.g. an earlier night aborted mid-wipe leaving scholar/grant/opportunity tables gutted, or a WHERE/flag change filters everything — so fillFn 'succeeds' and rebuildAliasedIndex atomically swaps the alias onto an EMPTY index (alias-swap.ts:279-280). Only THEN does main() run the smoke checks (index.ts:754-759), which throw post-swap: the step fails and pages, but the empty index is already the live alias target. Worse, a merely-shrunken corpus (say 90% of publications lost to a partial upstream night that still exited 0) swaps live and PASSES smoke entirely — the assertions probe specific field invariants ('Neoplasms' present, meshDescriptorUi >= 2500), never a doc-count delta versus the previous version.
- **Blast radius:** Site-wide search (people / publications / funding / opportunities tabs, autocomplete) serves empty or gutted results to every user until an operator manually repoints the alias at the retained prior version (docs/search.md rollback) or the corpus is repaired and the step reruns. The step's own failure also aborts Revalidate, so ISR caches go stale on top.
- **Recommendation:** Run the smoke assertions against the NEW concrete index BEFORE swapAlias (pass concreteIndex into the assert fns), and add a pre-swap guard comparing the new doc count to the current alias target's count (abort if new < N% of old, and hard-abort on 0 docs for a source that previously had docs). The retained prior version already makes this cheap to enforce.

### [HIGH] Every nightly step is Catch->SNS->Fail, so one flaky mid-chain source aborts the closing search:index + revalidate — diverging from orchestrate.ts's per-source failure isolation

- **Where:** `cdk/lib/etl-stack.ts:701` · category: partial-failure
- **Verification:** CONFIRMED (1 independent adversarial verifier, 1 affirmed)
- **Failure scenario:** buildStep (cdk/lib/etl-stack.ts:684-705) attaches the same Retry x2 + Catch->SnsPublish->Fail to EVERY step, and buildStateMachine (:730-734) chains them strictly sequentially. The design intent in etl/orchestrate.ts is the opposite: only ED aborts the cascade (orchestrate.ts:52-58); all other sources are failure-isolated (:63-123 'Per-source failure does not stop sibling sources'), and the search reindex 'runs unconditionally' (:127) with revalidate always attempted (:157). In the deployed machine, a single transient failure in ANY of ~10 external-source steps (Reciter, Asms, Infoed prod, Coi, Jenzabar, Dynamodb, Identity, Tools...) kills SearchIndexNightly and RevalidateNightly.
