## Item 5 — migration

`prisma/migrations/20260621170000_add_biosketch_impersonated_cwid/` — additive, nullable
`ADD COLUMN impersonated_cwid VARCHAR(32) NULL`. Apply via the **`sps-migrate-<env>`** task on deploy
(staging, then prod). Existing rows read NULL; their `created_by_cwid` predates this change (it held
the effective/overlay cwid for any old impersonated generation), so the "Generated by …" line is only
strictly accurate for rows written after this lands — acceptable for a brand-new, staging-only feature.

## Item 6 — operator runbook (the DB grant)

`app_ro` (the `db.read` role) is provisioned **out-of-band by a DBA** (the seeder manages only
`sps_bootstrap`/`sps_migrate` and tightens `app_rw`; it never touches `app_ro`). `sps_bootstrap`
holds `CREATE, ALTER, INSERT … WITH GRANT OPTION` on `scholars_audit.*` but **not SELECT**, so it
cannot issue this grant. It must be run by the RDS **master/DBA** on **staging and prod**:

```sql
-- Grant the read role SELECT on the audit table (read-only display of /history).
-- app_rw stays INSERT-only and no role gets UPDATE/DELETE → append-only/tamper-evidence intact.
GRANT SELECT ON `scholars_audit`.`manual_edit_audit` TO 'app_ro'@'%';
FLUSH PRIVILEGES;
```

(Use the env's actual `app_ro` host pattern — `%` on Aurora; staging app users are `@'%'`.)

**Then codify it** so it can't silently regress, in a small follow-up PR (kept separate so the
verify-grants gate doesn't red before the grant lands):
1. `scripts/verify-db-grants.ts` — add to the `app-ro` golden:
   `` "GRANT SELECT ON `scholars_audit`.`manual_edit_audit` TO `app_ro`@`%`" ``
2. `scripts/sql/audit-log.sql` — document the read-role SELECT alongside the existing `app_rw` INSERT template.

**Ordering:** apply the DBA grant on both envs FIRST, then merge the golden update (else the
`sps-verify-grants-<env>` task fails until the grant exists). Once granted, the app-side guard becomes
a belt-and-suspenders fallback and `/history` shows rows again.
